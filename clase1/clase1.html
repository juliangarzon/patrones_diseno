SOLID: la base del buen diseño orientado a objetos

Conjunto de 5 principios que promueven código mantenible, escalable y comprensible, formulados por Robert C. Martin ("Uncle Bob")

S
Single Responsibility
Una clase, una responsabilidad
📝 Single Responsibility Principle (SRP)

Explicación: Una clase debe tener una sola responsabilidad.

💡 Metáfora: Como un cajón: si guardas todo ahí, encontrar algo es un caos.
O
Open/Closed
Abierta a extensión, cerrada a modificación
🔒 Open/Closed Principle (OCP)

Explicación: Las clases deben poder extenderse sin modificarse.

💡 Metáfora: Como agregar una toma triple sin romper la pared.
L
Liskov Substitution
Las subclases deben poder sustituir a la clase padre
🔄 Liskov Substitution Principle (LSP)

Explicación: Una subclase debe comportarse como su clase base.

💡 Metáfora: Si un gato es un animal, debería poder usarse donde se espera un animal sin romper nada.
I
Interface Segregation
No obligues a implementar lo que no se necesita
🎛️ Interface Segregation Principle (ISP)

Explicación: No forzar a implementar lo que no se usa.

💡 Metáfora: Como un control remoto con 30 botones que solo usas 4.
D
Dependency Inversion
Depende de abstracciones, no de implementaciones
🔌 Dependency Inversion Principle (DIP)

Explicación: Depender de abstracciones, no implementaciones concretas.

💡 Metáfora: Como conectar un cargador USB sin importar la marca del cargador.

🤖
Patrones Creacionales
🤖 Patrones Creacionales
Se encargan de la creación de objetos de manera flexible y reutilizable.

Ejemplos: Singleton, Factory, Builder, Abstract Factory

Problema que resuelven: Cómo crear objetos sin especificar sus clases exactas
🏗️
Patrones Estructurales
🏗️ Patrones Estructurales
Se ocupan de la composición de clases y objetos para formar estructuras más grandes.

Ejemplos: Adapter, Decorator, Facade, Composite

Problema que resuelven: Cómo ensamblar objetos y clases en estructuras flexibles
🎭
Patrones de Comportamiento
🎭 Patrones de Comportamiento
Se centran en la comunicación entre objetos y la asignación de responsabilidades.

Ejemplos: Observer, Strategy, Command, State

Problema que resuelven: Cómo los objetos interactúan y distribuyen responsabilidades

Antipatrones: cuando el diseño se vuelve un problema

⚠️ "Mala solución popular"

Parece correcta, pero genera más problemas

Suelen aparecer por:

  • 🚀 Falta de experiencia
  • ⏰ Presión por entregar rápido
  • 📋 Copiar soluciones sin entenderlas

Ejemplos comunes:

  • 🌐 God Object
  • 🍝 Spaghetti Code
  • 🌋 Lava Flow
  • 📝 Copy-paste programming

God Object: el objeto que lo quiere controlar todo

⚠️ Características

  • • Una sola clase con muchas responsabilidades
  • • Viola SRP (Single Responsibility Principle)
  • • Es difícil de mantener, probar y extender
  • • Muy común en sistemas sin modularización
class Sistema {
  void registrarUsuario() {...}
  void generarReporte() {...}
  void enviarCorreo() {...}
  void procesarPago() {...}
  void validarDatos() {...}
  void conectarDB() {...}
  void renderizarUI() {...}
}
🚨 Una clase haciendo todo = Problema

Spaghetti Code: todo conectado con todo

🍝 Características

  • • Código sin estructura clara
  • • Flujo de ejecución difícil de seguir
  • • Cambiar una parte puede romper muchas otras
  • • Lógica de negocio mezclada

Lava Flow: código que ya nadie entiende, pero sigue ahí

🌋 Características

  • • Restos de código antiguo que ya no se usa
  • • Nadie se atreve a borrar
  • • Ocupa espacio y genera confusión
  • • Puede causar errores inesperados
